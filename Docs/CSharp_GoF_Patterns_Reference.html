Behavioural Patterns
🔀
1. Strategy
Replace if/switch on type — each variant becomes its own class implementing a shared interface.
Behavioural OCP 🔴 Most common in tests
Bad — switch grows with every type
public decimal GetDiscount(string type,
  decimal total)
{
  if (type == "VIP")    return total * .30m;
  if (type == "Member") return total * .15m;
  if (type == "New")    return total * .05m;
  // add type = EDIT THIS METHOD
  return 0m;
}
Fixed — Strategy pattern
public interface IDiscountStrategy
{
  string  CustomerType { get; }
  decimal Calculate(decimal total);
}
public class VipDiscount : IDiscountStrategy
{
  public string  CustomerType => "VIP";
  public decimal Calculate(decimal t) => t*.30m;
}
// add type = add 1 class, touch nothing

public class DiscountService
{
  private readonly Dictionary<string,
    IDiscountStrategy> _map;
  public DiscountService(
    IEnumerable<IDiscountStrategy> s)
    => _map = s.ToDictionary(x=>x.CustomerType);
  public decimal GetDiscount(string t,
    decimal total)
    => _map.TryGetValue(t, out var s)
       ? s.Calculate(total) : 0m;
}
🔍 Smell that triggers this pattern
  • Search: if (type == "X") or switch(type) in service classes
  • Adding a new type requires editing an existing method
  • Same switch duplicated in multiple methods
// Program.cs DI registration builder.Services.AddScoped<IDiscountStrategy, VipDiscount>(); builder.Services.AddScoped<IDiscountStrategy, MemberDiscount>(); // IEnumerable<IDiscountStrategy> auto-collects all registrations
✓ Use when
  • Multiple variants of the same algorithm
  • New variants added frequently
  • if/switch on a type string
✗ Don't use when
  • Only 2 variants that never change
  • Simple null-check or boolean guard
🗄
2. Repository
Abstract data access — business logic never knows what database engine is used.
Structural DIP 🔴 Every test has this
Bad — DB inside business logic
public class OrderService
{
  public Order GetOrder(int id)
  {
    var conn = new SqlConnection("...");
    return conn.QueryFirst<Order>(
      $"SELECT * FROM Orders WHERE Id={id}");
    // SQL injection + tight coupling
  }
}
Fixed — Repository pattern
// interface in domain layer
public interface IOrderRepository
{
  Task<Order?> GetByIdAsync(int id);
  Task          SaveAsync(Order order);
  Task          DeleteAsync(int id);
}

// implementation in infra layer
public class SqlOrderRepository
  : IOrderRepository
{
  public async Task<Order?> GetByIdAsync(int id)
  {
    await using var conn = new SqlConnection(_cs);
    return await conn.QueryFirstOrDefaultAsync<Order>(
      "SELECT * FROM Orders WHERE Id=@Id",
      new { Id = id });
  }
}

// business logic — no SQL anywhere
public class OrderService
{
  private readonly IOrderRepository _repo;
  public OrderService(IOrderRepository r)
    => _repo = r;
  public Task<Order?> GetOrderAsync(int id)
    => _repo.GetByIdAsync(id);
}
🔍 Smell that triggers this pattern
  • Search: new SqlConnection( inside a service class
  • Search: $"SELECT ... {variable}" — SQL injection
  • Cannot unit test service without a real database running
  • Changing from Dapper → EF Core means editing service logic
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>(); // test: new Mock<IOrderRepository>() — no DB needed
✓ Use when
  • Any class reads/writes a database
  • Need to swap DB engine or mock in tests
  • Business logic must stay DB-agnostic
✗ Don't use when
  • Simple CRUD script, no business logic
  • Using Event Sourcing (use event store instead)
🏭
3. Factory / Factory Method
Encapsulate object creation — callers ask for an object without knowing the concrete type.
Creational OCP 🟡 Common in tests
Bad — creation scattered in callers
// caller must know all concrete types
IPaymentGateway gw;
if (method == "stripe")
  gw = new StripeGateway("sk_live_");
else if (method == "paypal")
  gw = new PayPalGateway("id", "sec");
// duplicated in every caller
Fixed — Factory pattern
public interface IPaymentGatewayFactory
{
  IPaymentGateway Create(string method);
}

public class PaymentGatewayFactory
  : IPaymentGatewayFactory
{
  private readonly IConfiguration _cfg;
  public PaymentGatewayFactory(IConfiguration c)
    => _cfg = c;

  public IPaymentGateway Create(string method)
    => method switch
    {
      "stripe" => new StripeGateway(
        _cfg["Stripe:Key"]!),
      "paypal" => new PayPalGateway(
        _cfg["PayPal:Id"]!, _cfg["PayPal:Secret"]!),
      _ => throw new NotSupportedException(method)
    };
}

// caller — knows nothing about concrete types
var gw = _factory.Create(method);
await gw.ChargeAsync(amount, token);
🔍 Smell that triggers this pattern
  • Same if/switch + new ConcreteClass() block copied in multiple callers
  • Caller must know constructor args of every concrete type
  • Adding a new payment method → edit every caller
builder.Services.AddSingleton<IPaymentGatewayFactory, PaymentGatewayFactory>();
✓ Use when
  • Object creation logic is complex or conditional
  • Multiple callers need same creation logic
  • Creation requires config / secrets
✗ Don't use when
  • Simple new Foo() with no variants
  • DI container already handles creation
🎁
4. Decorator
Add behaviour without editing — wrap an existing class with the same interface to layer on cross-cutting concerns.
Structural OCP 🟡 Common in tests
Bad — logging tangled in business logic
public class OrderRepository : IOrderRepository
{
  public async Task SaveAsync(Order o)
  {
    _logger.LogInformation("Saving {Id}", o.Id);
    // actual save logic
    await _db.ExecuteAsync(...);
    _logger.LogInformation("Saved {Id}", o.Id);
    // logging = separate concern mixed in
  }
}
Fixed — Decorator wraps the real repo
// real implementation — clean, no logging
public class SqlOrderRepository
  : IOrderRepository
{
  public async Task SaveAsync(Order o)
    => await _db.ExecuteAsync(...);
}

// decorator adds logging transparently
public class LoggingOrderRepository
  : IOrderRepository
{
  private readonly IOrderRepository _inner;
  private readonly ILogger _log;
  public LoggingOrderRepository(
    IOrderRepository inner, ILogger<
    LoggingOrderRepository> log)
  { _inner = inner; _log = log; }

  public async Task SaveAsync(Order o)
  {
    _log.LogInformation("Saving {Id}", o.Id);
    await _inner.SaveAsync(o);  // delegate
    _log.LogInformation("Saved {Id}", o.Id);
  }
}
// Scrutor NuGet makes DI easy:
// services.Decorate<IOrderRepository,
//                  LoggingOrderRepository>();
🔍 Smell that triggers this pattern
  • Logging, caching, timing, retry code mixed into core business methods
  • Same cross-cutting concern copy-pasted across multiple classes
  • Adding caching means editing the real implementation
// Scrutor package — clean decorator registration services.AddScoped<IOrderRepository, SqlOrderRepository>(); services.Decorate<IOrderRepository, LoggingOrderRepository>(); services.Decorate<IOrderRepository, CachingOrderRepository>();
✓ Use when
  • Adding logging / caching / retry without editing
  • Cross-cutting concern applies to one interface
  • You want layers you can toggle
✗ Don't use when
  • Only one implementation will ever exist
  • Middleware pipeline already handles it
🔄
5. State
Status transition logic — each status owns its own behaviour and valid transitions.
Behavioural OCP 🟡 Common in tests
Bad — switch in every method
public void Process(Order o)
{
  switch (o.Status) {
    case "Pending":   Validate(o);  break;
    case "Confirmed": Charge(o);    break;
    case "Paid":      Ship(o);      break;
  }
}
public string GetLabel(Order o)
{
  // same switch again — shotgun surgery
  if (o.Status == "Pending") return "Awaiting...";
}
Fixed — State pattern
public interface IOrderState
{
  string StatusName { get; }
  string Label      { get; }
  Order  Process(Order order);
}

public class PendingState : IOrderState
{
  public string StatusName => "Pending";
  public string Label      => "Awaiting confirmation";
  public Order Process(Order o)
  { Validate(o); o.Status="Confirmed"; return o; }
}
// each status = 1 class, 1 reason to change

public class OrderService
{
  private readonly Dictionary<string,IOrderState> _states;
  public Order  Process(Order o)
    => GetState(o).Process(o);  // no switch!
  public string GetLabel(Order o)
    => GetState(o).Label;         // no switch!
}
🔍 Smell that triggers this pattern
  • Search: switch(order.Status) in more than one method
  • Adding a new status requires editing multiple switch blocks (shotgun surgery)
  • Status + behaviour are not co-located — hard to reason about
builder.Services.AddScoped<IOrderState, PendingState>(); builder.Services.AddScoped<IOrderState, ConfirmedState>(); builder.Services.AddScoped<IOrderState, PaidState>();
✓ Use when
  • Object has multiple statuses with different behaviour
  • Same switch repeated in many methods
  • Status transitions need validation
✗ Don't use when
  • Only 2 states with trivial logic
  • Simple boolean flag is enough
📡
6. Observer / Domain Event
Decouple side effects — when something happens, notify subscribers without the source knowing who's listening.
Behavioural SRP 🟡 Common in tests
Bad — side effects hardcoded
public void PlaceOrder(Order order)
{
  _repo.Save(order);
  // side effects hardcoded — SRP broken
  _mailer.Send(order.Email, "Confirmed");
  _inventory.Reduce(order.Items);
  _analytics.Track(order);
  // add new side effect = edit THIS method
}
Fixed — Domain Event pattern
// 1. define the event
public record OrderPlacedEvent(Order Order);

// 2. handlers — one per side effect
public class SendConfirmationEmail
  : INotificationHandler<OrderPlacedEvent>
{
  public Task Handle(OrderPlacedEvent e,
    CancellationToken ct)
    => _mailer.SendAsync(e.Order.Email, "Confirmed");
}
// add new side effect = add new handler class

// 3. source — only publishes, knows nothing
public async Task PlaceOrderAsync(Order order)
{
  _repo.Save(order);
  // publish — all handlers called automatically
  await _mediator.Publish(
    new OrderPlacedEvent(order));
}
🔍 Smell that triggers this pattern
  • Core method calls 3+ side-effect services (email, inventory, analytics)
  • Adding a new side effect means editing the core method
  • Side effects are not directly related to the core action
// MediatR auto-discovers all INotificationHandler implementations builder.Services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(typeof(Program).Assembly));
✓ Use when
  • Core action triggers multiple independent side effects
  • Side effects may change independently of core
  • Cross-service fan-out needed
✗ Don't use when
  • Only 1 side effect, tightly coupled to core
  • Simple callback is enough
🔌
7. Adapter
Wrap external SDKs — translate a third-party API into your own interface so your domain never imports vendor code.
Structural DIP 🔴 Every infra test
Bad — vendor SDK in business logic
public class PaymentService
{
  public async Task ChargeAsync(decimal amt)
  {
    // Stripe SDK directly in domain — DIP broken
    var stripe = new StripeClient("sk_live_");
    await stripe.Charges.CreateAsync(
      new ChargeCreateOptions
      { Amount=(long)(amt*100),Currency="usd" });
    // Stripe leaks into every layer
  }
}
Fixed — Adapter wraps Stripe
// domain interface — no vendor types
public interface IPaymentGateway
{
  Task<string> ChargeAsync(decimal amount,
    string currency);
}

// adapter — lives in infra layer only
public class StripePaymentAdapter
  : IPaymentGateway
{
  private readonly StripeClient _client;
  public StripePaymentAdapter(
    IOptions<StripeSettings> opts)
    => _client = new StripeClient(opts.Value.Key);

  public async Task<string> ChargeAsync(
    decimal amount, string currency)
  {
    var c = await _client.Charges.CreateAsync(
      new ChargeCreateOptions
      { Amount=(long)(amount*100), Currency=currency});
    return c.Id;  // return domain type, not Stripe type
  }
}
// swap to PayPal: class PayPalAdapter : IPaymentGateway
🔍 Smell that triggers this pattern
  • Search: using Stripe; / using Twilio; / using SendGrid; in domain/service files
  • Vendor SDK types (StripeClient, SmtpClient) appear in business logic
  • Swapping vendor requires editing business logic files
builder.Services.Configure<StripeSettings>(cfg.GetSection("Stripe")); builder.Services.AddScoped<IPaymentGateway, StripePaymentAdapter>(); // test: new Mock<IPaymentGateway>() — no Stripe account needed
✓ Use when
  • Any third-party SDK (Stripe, Twilio, SendGrid)
  • Infrastructure concerns (file, queue, cache)
  • Want to swap vendor without touching business code
✗ Don't use when
  • SDK is already perfectly aligned with your interface
  • Internal utility with no external dependency
📨
8. Mediator (MediatR / CQRS)
Decouple sender from handler — controller sends a command/query, MediatR routes it to the correct handler.
Behavioural SRP Lead Level
Bad — fat controller / fat service
public class OrderController
{
  // controller directly calls service = coupled
  private readonly OrderService    _orders;
  private readonly InventoryService _inv;
  private readonly NotificationService _notif;

  public IActionResult Place(Order o)
  {
    _orders.Save(o);
    _inv.Reserve(o.Items);
    _notif.Notify(o.CustomerId);
    return Ok();
  }
}
Fixed — MediatR CQRS
// command (write operation)
public record PlaceOrderCommand(Order Order)
  : IRequest<Guid>;

// handler — single responsibility
public class PlaceOrderHandler
  : IRequestHandler<PlaceOrderCommand, Guid>
{
  public async Task<Guid> Handle(
    PlaceOrderCommand cmd,
    CancellationToken ct)
  {
    await _repo.SaveAsync(cmd.Order);
    await _mediator.Publish(
      new OrderPlacedEvent(cmd.Order), ct);
    return cmd.Order.Id;
  }
}

// query (read operation)
public record GetOrderQuery(Guid Id)
  : IRequest<OrderDto?>;

// controller — 1 dependency only
public class OrderController
{
  private readonly IMediator _mediator;
  public Task<IActionResult> Place(Order o)
    => _mediator.Send(new PlaceOrderCommand(o));
}
🔍 Smell that triggers this pattern
  • Controller constructor injects 3+ service dependencies
  • Service method is long and mixes read and write logic
  • Cross-cutting concerns (logging, validation) applied inconsistently — use IPipelineBehavior
  • Hard to test controller in isolation from services
builder.Services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(typeof(Program).Assembly)); // IPipelineBehavior for cross-cutting (logging, validation, timing): builder.Services.AddTransient(typeof(IPipelineBehavior<,>), typeof(LoggingBehavior<,>));
✓ Use when
  • Controllers have 3+ service dependencies
  • Want consistent cross-cutting via pipeline
  • CQRS separation of read/write paths
✗ Don't use when
  • Simple CRUD with no business logic
  • Small project where indirection adds friction
