S — Single Responsibility Principle
🏛
God Class
SRP · Single Responsibility
🔴 CriticalS
Bad code
public class UserManager {
// validate + save DB + send email + generate PDF
public void Process(User user) {
// 150+ lines of mixed concerns
}
}
  • File over 200 lines — near-certain SRP violation
  • Class name ends in Manager, Handler, Helper, Utils
  • Constructor takes 5+ dependencies
  • Multiple private method groups with unrelated concerns
✓ Fix: Extract Class — Service / Repository / Mailer
🔀
Mixed Abstraction Levels
SRP · Single Responsibility
🔴 CriticalS
Bad code
public void PlaceOrder(Order order) {
// business rule
if (order.Total <= 0) throw...;
// raw SQL inside business logic
var conn = new SqlConnection("...");
conn.Execute("INSERT...");
new SmtpClient().Send(...);
}
  • Business logic and SQL/HTTP/SMTP in same method
  • Method does "what to do" and "how to do it technically"
  • Function longer than 20–30 lines
✓ Fix: Extract Method + separate infra classes
O — Open / Closed Principle
🔀
if / else Chain on Type
OCP · Open/Closed
🔴 CriticalO
Bad code
public decimal GetDiscount(string type) {
if (type == "VIP") return 0.30m;
else if (type == "Member") return 0.15m;
else if (type == "New") return 0.05m;
// add type = EDIT THIS FILE
return 0m;
}
  • Search: if (x == "...") or switch(type) in service classes
  • Every new business type forces editing this method
  • Same switch duplicated in multiple methods
✓ Fix: Strategy pattern — one class per type
🎛
switch on Action / Status
OCP · Open/Closed
🔴 CriticalO
Bad code
switch (order.Status) {
case "Pending": ValidateOrder(); break;
case "Confirmed": ChargePayment(); break;
case "Paid": ShipOrder(); break;
// add status = edit THIS + other methods
}
  • Same switch in multiple methods (Process, GetLabel, GetIcon)
  • Adding new status → shotgun surgery across files
  • Status logic scattered, not encapsulated
✓ Fix: State pattern — each status owns its transitions

LSP · Liskov Substitution
🔴 CriticalL

public class ReadOnlyList : List {

1
SQL Injection → Parameterized Queries
String interpolation in SQL = critical security vulnerability. Any user input passed directly into a query string can delete your database.
🔴 Critical — SecurityDIP adjacent
Bad — SQL injection risk
public void SaveInvoice(Invoice inv)
{
  var conn = new SqlConnection(
    "Server=localhost;Password=1234");
  conn.Open();
  // ⚠ string interpolation = injection
  var sql = $"INSERT INTO Invoices "
          + $"VALUES ('{inv.Customer}',"
          + $"'{inv.Amount}', '{inv.Date}')";
  conn.Execute(sql);
}
Fixed — parameterized query
public async Task SaveInvoiceAsync(Invoice inv)
{
  await using var conn = new SqlConnection(
    _config.GetConnectionString("Default"));
  // ✓ named parameters — injection impossible
  const string sql =
    "INSERT INTO Invoices "
    "(Customer, Amount, Date) "
    "VALUES (@Customer, @Amount, @Date)";
  await conn.ExecuteAsync(sql, new
  { inv.Customer, inv.Amount, inv.Date });
}
🔍 How to find it
  • Search: $"... {variable} ..." inside any SQL string
  • Search: "SELECT" + variable or "INSERT" + variable
  • Any Execute() / Query() call with a built string, not a const
  • No @ParameterName in the SQL and no anonymous object passed as second arg
Rule
SQL strings must be const string — never built with interpolation or concatenation. User input goes in the parameter object only.
2
Tight Coupling → IInvoiceRepository + IEmailSender Injected
Business logic directly instantiates SqlConnection, SmtpClient — impossible to test, impossible to swap. DIP broken.
🔴 Critical — DIPSRP
Bad — hardcoded dependencies
public class InvoiceService
{
  // ⚠ new = tightly coupled
  private readonly SqlConnection _db = new SqlConnection("...");
  private readonly SmtpClient _smtp = new SmtpClient("smtp.host");
}
Fixed — injected interfaces
public interface IInvoiceRepository
{
  Task SaveAsync(Invoice invoice);
  Task<Invoice?> GetByIdAsync(int id);
}
public interface IEmailSender
{
  Task SendAsync(string to, string subject, string body);
}
// ✓ depends on abstractions only
public class InvoiceService
{
  private readonly IInvoiceRepository _repo;
  private readonly IEmailSender _email;
  public InvoiceService(IInvoiceRepository repo, IEmailSender email)
  { _repo = repo; _email = email; }
}
🔍 How to find it
  • Search: = new ConcreteClass() as a field initializer
  • Search: new SqlConnection / new SmtpClient / new HttpClient inside class bodies
  • Constructor has zero parameters but class does I/O
Principle: DIP — Dependency Inversion
High-level modules (InvoiceService) must not depend on low-level modules (SqlConnection). Both depend on abstractions (IInvoiceRepository). Interfaces live in the domain layer. Concrete classes live in the infrastructure layer.
3
SRP Violation → Validation / Pricing / Persistence / Notification Split
One class doing four jobs. Every change to pricing, DB schema, email template, or validation rules touches the same file.
🔴 Critical — SRPExtract Class
Bad — one method, four concerns
public class InvoiceManager  // ⚠
{
  public void Process(Invoice inv)
  {
    // 1. validate
    if (inv.Amount <= 0) throw new Exception("bad");
    // 2. calculate price
    inv.Tax = inv.Amount * 0.07m;
    // 3. save to DB
    var conn = new SqlConnection(...); conn.Execute($"INSERT...");
    // 4. send email
    new SmtpClient().Send(...);
  }
}
Fixed — each class has one job
public class InvoiceValidator
{
  public void Validate(Invoice inv)
  { if (inv.Amount <= 0) throw new ArgumentException("Amount must be positive"); }
}
public class InvoicePricer
{
  public void ApplyTax(Invoice inv)
  { inv.Tax = inv.Amount * TaxRate; inv.Total = inv.Amount + inv.Tax; }
}
// 3. persistence → IInvoiceRepository
// 4. notification → IEmailSender
// orchestrator — no logic, just coordination
public class InvoiceService
{
  public async Task ProcessAsync(Invoice inv)
  {
    _validator.Validate(inv);
    _pricer.ApplyTax(inv);
    await _repo.SaveAsync(inv);
    await _email.SendAsync(...);
  }
}
🔍 How to find it
  • Ask: "who would ask me to change this class?" — PM (logic), DBA (SQL), designer (email) = 3 owners = SRP broken
  • Method contains comment blocks labelling different concerns (// validate, // save, // email)
  • Class name ends in Manager, Processor, Handler
  • File is over 150 lines or method over 30 lines
Principle: SRP — Single Responsibility
A class has one reason to change. Split by concern: Validator / Pricer / Repository / Mailer. Each class has exactly one owner.
4
Magic Literals → Constants + Configuration
0.07m, 30, "no-reply@app.com" scattered inline — no one knows what they mean. Change requires grep across the whole codebase.
🟡 High — ReadabilityDRY
Bad — magic everywhere
public void Process(Invoice inv)
{
  inv.Tax = inv.Amount * 0.07m; // what rate?
  inv.DueDate = inv.Date.AddDays(30); // why 30?
  if (inv.Amount > 10000) _email.Send("boss@co.com");
}
Fixed — named constants + config
private const decimal TaxRate          = 0.07m;
private const int     PaymentTermsDays = 30;
private const decimal LargeInvoiceLimit= 10_000m;

public void Process(Invoice inv)
{
  inv.Tax     = inv.Amount * TaxRate;
  inv.DueDate = inv.Date.AddDays(PaymentTermsDays);
  if (inv.Amount > LargeInvoiceLimit)
    _email.Send(_settings.AlertEmail);
}
🔍 How to find it
  • Any decimal/int literal that is not 0 or 1 inside business logic
  • Literal strings for emails, roles, status values, queue names
  • Same number appears in 2+ places
Rule: Two categories
Business constants → private const. Environment config → appsettings.json bound via IOptions<T>.
5

1. Magic Cache Key Strings
Inline string keys scattered across classes — typo = silent cache miss with no error.
🔴 CriticalDRY violationSilent bug
Bad — inline string keys
public Product? GetProduct(int id)
{
  var key = "product_" + id;  // ⚠ magic string
  if (_cache.TryGetValue(key, out Product? p)) return p;
  p = _repo.Get(id); _cache.Set(key, p); return p;
}
public void UpdateProduct(Product p)
{
  _repo.Update(p);
  _cache.Remove("products_" + p.Id);  // ⚠ different format!
}
Fixed — typed key factory
public static class CacheKeys
{
  public static string Product(int id) => $"products:v1:{id}";
  public static string ProductList() => "products:v1:all";
  public static string UserProfile(Guid userId) => $"users:v1:{userId}";
}
public void UpdateProduct(Product p)
{
  _repo.Update(p);
  _cache.Remove(CacheKeys.Product(p.Id));  // ✓ same method
}
🔍 How to find this smell
  • Search: "product_" + id or $"cache_{variable}" inline in methods

Key naming convention
Format: entity:version:id — e.g. "products:v1:42". Include a version so you can bust all entries by bumping v1 → v2.

2. Missing or Wrong Expiry
No TTL = stale data forever. Magic number TTL = impossible to tune without redeploying.
🟡 HighStale dataMemory leak
Bad — no expiry / magic number
_cache.Set(key, p);  // ⚠ no expiry — lives forever
_cache.Set(key, p, TimeSpan.FromSeconds(300));  // ⚠ what is 300?
Fixed — named expiry from config
public static class CacheExpiry
{
  public static readonly TimeSpan Product     = TimeSpan.FromMinutes(5);
  public static readonly TimeSpan UserSession = TimeSpan.FromMinutes(15);
  public static readonly TimeSpan ReferenceData= TimeSpan.FromHours(1);
}
_cache.Set(key, p, new MemoryCacheEntryOptions
{
  AbsoluteExpirationRelativeToNow = CacheExpiry.Product,
  Priority = CacheItemPriority.Normal
});
Best practice for sliding
Always combine: SlidingExpiration = 2min + AbsoluteExpirationRelativeToNow = 10min.

3. Cache Stampede (Dog-piling)
Cache expires → 50 concurrent requests all miss → all hit DB simultaneously → DB collapses.
🔴 CriticalRace conditionDB overload
Bad — no lock, all threads hit DB
// ⚠ no locking — all 50 threads miss simultaneously on expiry
if (_cache.TryGetValue(key, out Product? cached)) return cached;
var product = await _repo.GetAsync(id);  // all 50 hit DB at once
Fixed — SemaphoreSlim per key
private readonly ConcurrentDictionary<string, SemaphoreSlim> _locks = new();

// fast path — no lock needed
if (_cache.TryGetValue(key, out Product? cached)) return cached;
// slow path — only ONE thread loads
var sem = _locks.GetOrAdd(key, _ => new SemaphoreSlim(1, 1));
await sem.WaitAsync();
try {
  if (_cache.TryGetValue(key, out cached)) return cached;  // double-check
  var product = await _repo.GetAsync(id);  // 1 DB call
  _cache.Set(key, product, CacheExpiry.Product); return product;
} finally { sem.Release(); }
Alternative: GetOrCreate
IMemoryCache.GetOrCreateAsync(key, entry => { ... return _repo.GetAsync(id); }) — handles the lock internally for simple cases.

4. Cache-Aside Pattern (Lazy Loading)
The correct standard pattern — check cache, miss → load from source → populate cache → return.
✓ Best PracticeStandard pattern
Bad — mixing cache + business logic
string key = "price_" + productId;  // ⚠ magic string
if (_cache.TryGetValue(key, out decimal cached)) return cached;
// business logic buried after cache logic
_cache.Set(key, price, TimeSpan.FromSeconds(300));  // ⚠ magic number
Fixed — clean cache-aside
return await _cache.GetOrCreateAsync(
  CacheKeys.Product(productId),
  async entry =>
  {
    entry.AbsoluteExpirationRelativeToNow = CacheExpiry.Product;
    entry.Priority = CacheItemPriority.High;
    var product = await _repo.GetAsync(productId);
    return product.BasePrice * (1 - product.Discount);
  })!;
Why GetOrCreateAsync beats manual TryGetValue
It handles the check-load-set pattern in one call. The factory is only invoked on a cache miss.

5. Missing Cache Invalidation

7. Blocking Async in Cache Methods
.Result / .Wait() inside cache methods cause deadlocks in ASP.NET Core.
🔴 CriticalDeadlock riskThread starvation
Bad — sync blocking async cache

  async e => await _repo.GetAsync(id))
  .Result;   // ← deadlock in ASP.NET Core!
Fixed — async all the way up

{
  return await _cache.GetOrCreateAsync(
    CacheKeys.Product(id),
    async entry =>
    {
      entry.AbsoluteExpirationRelativeToNow = CacheExpiry.Product;
      return await _repo.GetAsync(id);  // ✓ awaited
    });
}
Never use .Result, .Wait(), .GetAwaiter().GetResult() on async cache methods in ASP.NET Core
async/await must flow from the controller all the way down to the repository. No exceptions.

8. Generic GetOrSetAsync Helper
The same check-load-set pattern repeated in every method — extract to a reusable generic helper.
🟡 HighDRY violationExtract Method fix
Bad — repeated in every method
// same 5-line pattern in every service method
var key = CacheKeys.Product(id);
if (_cache.TryGetValue(key, out Product? p)) return p;
p = await _repo.GetAsync(id);
_cache.Set(key, p, CacheExpiry.Product); return p;
Fixed — generic GetOrSetAsync
public static async Task<T?> GetOrSetAsync<T>(
  this IMemoryCache cache, string key,
  Func<Task<T?>> factory, TimeSpan? expiry = null)
  where T : class
{
  if (cache.TryGetValue(key, out T? cached)) return cached;
  var value = await factory();
  if (value is not null) cache.Set(key, value, expiry ?? TimeSpan.FromMinutes(5));
  return value;
}
// usage — one clean line per method
public Task<Product?> GetProductAsync(int id)
  => _cache.GetOrSetAsync(CacheKeys.Product(id), () => _repo.GetAsync(id), CacheExpiry.Product);
DRY principle applied to caching
The check-load-set pattern is knowledge — it belongs in one place. The generic helper enforces consistent null-guarding and expiry defaults.

public class LoanService
{
    private const int     MinAge          = 18;
    private const int     MinCreditScore  = 650;
    private const decimal MinIncomeRatio  = 0.3m;
    private const decimal MaxLoanAmount   = 1_000_000m;

    public string ApproveLoan(User user, LoanRequest request)
    {
        // guard clauses — fail fast, flat structure
        if (user    == null)   return "User not found";
        if (!user.IsActive)    return "Inactive user";
        if (user.Age < MinAge) return "Underage";

        if (request == null)   return "No request";
        if (request.Amount <= 0)            return "Invalid amount";
        if (request.Amount > MaxLoanAmount) return "Over limit";

        // business rules — now readable
        if (!HasSufficientCreditScore(user))      return "Low credit";
        if (!HasSufficientIncome(user, request))  return "Low income";

        return "Approved";
    }

    private bool HasSufficientCreditScore(User user)
        => user.CreditScore >= MinCreditScore;

    private bool HasSufficientIncome(User user, LoanRequest request)
        => user.Income >= request.Amount * MinIncomeRatio;
}
