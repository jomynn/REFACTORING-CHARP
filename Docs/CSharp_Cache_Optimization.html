1. Magic Cache Key Strings
Inline string keys scattered across classes — typo = silent cache miss with no error.
🔴 Critical DRY violation Silent bug
Bad — inline string keys
public class ProductService
{
  public Product? GetProduct(int id)
  {
    // ⚠ magic string — typo = silent miss
    var key = "product_" + id;
    if (_cache.TryGetValue(key, out Product? p))
      return p;
    p = _repo.Get(id);
    _cache.Set(key, p);
    return p;
  }

  public void UpdateProduct(Product p)
  {
    _repo.Update(p);
    // ⚠ different format — cache never busted!
    _cache.Remove("products_" + p.Id);
  }
}
Fixed — typed key factory
// single source of truth for all cache keys
public static class CacheKeys
{
  public static string Product(int id)
    => $"products:v1:{id}";

  public static string ProductList()
    => "products:v1:all";

  public static string UserProfile(Guid userId)
    => $"users:v1:{userId}";
}

public class ProductService
{
  public Product? GetProduct(int id)
  {
    // ✓ typed method — compile-time safe
    var key = CacheKeys.Product(id);
    if (_cache.TryGetValue(key, out Product? p))
      return p;
    p = _repo.Get(id);
    _cache.Set(key, p);
    return p;
  }

  public void UpdateProduct(Product p)
  {
    _repo.Update(p);
    // ✓ same method = guaranteed match
    _cache.Remove(CacheKeys.Product(p.Id));
  }
}
🔍 How to find this smell
  • Search: "product_" + id or $"cache_{variable}" inline in methods
  • Same entity cached with different key format in set vs remove
  • Key string duplicated in multiple classes
Key naming convention
Format: entity:version:id — e.g. "products:v1:42". Include a version so you can bust all entries by bumping v1 → v2. Never use raw concatenation — always use a static factory method.

2. Missing or Wrong Expiry
No TTL = stale data forever. Magic number TTL = impossible to tune without redeploying.
🟡 High Stale data Memory leak
Bad — no expiry / magic number
public Product? GetProduct(int id)
{
  var key = $"product:{id}";
  if (_cache.TryGetValue(key, out Product? p))
    return p;

  p = _repo.Get(id);

  // ⚠ no expiry — lives forever in memory
  _cache.Set(key, p);

  // OR: ⚠ magic number — what is 300?
  _cache.Set(key, p,
    TimeSpan.FromSeconds(300));

  return p;
}
Fixed — named expiry from config
// expiry constants — business-driven names
public static class CacheExpiry
{
  // how long a product is "fresh"
  public static readonly TimeSpan Product
    = TimeSpan.FromMinutes(5);

  // user sessions — shorter
  public static readonly TimeSpan UserSession
    = TimeSpan.FromMinutes(15);

  // reference data — rarely changes
  public static readonly TimeSpan ReferenceData
    = TimeSpan.FromHours(1);
}

public Product? GetProduct(int id)
{
  var key = CacheKeys.Product(id);
  if (_cache.TryGetValue(key, out Product? p))
    return p;

  p = _repo.Get(id);

  // ✓ named expiry — self-documenting
  _cache.Set(key, p,
    new MemoryCacheEntryOptions
    {
      AbsoluteExpirationRelativeToNow
        = CacheExpiry.Product,
      // evict under memory pressure
      Priority = CacheItemPriority.Normal
    });
  return p;
}
🔍 Expiry types — when to use each
  • AbsoluteExpiration — fixed point in time. Use for: tokens, scheduled data
  • AbsoluteExpirationRelativeToNow — TTL from now. Use for: most product/user data
  • SlidingExpiration — resets on access. Use for: sessions, user preferences. Never alone — pair with absolute to prevent infinite extension
Best practice for sliding
Always combine: SlidingExpiration = 2min + AbsoluteExpirationRelativeToNow = 10min. Sliding alone lets an active user cache live forever.

3. Cache Stampede (Dog-piling)
Cache expires → 50 concurrent requests all miss → all hit DB simultaneously → DB collapses.
🔴 Critical Race condition DB overload
Bad — no lock, all threads hit DB
public async Task<Product?>
  GetProductAsync(int id)
{
  var key = CacheKeys.Product(id);

  // ⚠ no locking — all 50 threads
  // miss simultaneously on expiry
  if (_cache.TryGetValue(key,
    out Product? cached)) return cached;

  // all 50 hit DB at once
  var product = await _repo.GetAsync(id);
  _cache.Set(key, product,
    CacheExpiry.Product);
  return product;
}
Fixed — SemaphoreSlim per key
// one semaphore per cache key
private readonly ConcurrentDictionary<string,
  SemaphoreSlim> _locks = new();

public async Task<Product?>
  GetProductAsync(int id)
{
  var key = CacheKeys.Product(id);

  // fast path — no lock needed
  if (_cache.TryGetValue(key,
    out Product? cached)) return cached;

  // slow path — only ONE thread loads
  var sem = _locks.GetOrAdd(key,
    _ => new SemaphoreSlim(1, 1));
  await sem.WaitAsync();
  try
  {
    // double-check after acquiring lock
    if (_cache.TryGetValue(key,
      out cached)) return cached;

    var product =
      await _repo.GetAsync(id);  // 1 DB call
    _cache.Set(key, product,
      CacheExpiry.Product);
    return product;
  }
  finally { sem.Release(); }
}
🔍 The double-check pattern explained
  • Fast path — check cache first with no lock (99% of requests return here)
  • Acquire semaphore — only one thread proceeds to DB at a time
  • Double-check — the thread that waited may now find the cache populated by the thread ahead
  • Load + set — exactly one DB call per cache miss, all others wait then read from cache
Alternative: GetOrCreate
IMemoryCache.GetOrCreateAsync(key, entry => { entry.AbsoluteExpirationRelativeToNow = CacheExpiry.Product; return _repo.GetAsync(id); }) — handles the lock internally for simple cases. Use SemaphoreSlim for fine-grained control.

4. Cache-Aside Pattern (Lazy Loading)
The correct standard pattern — check cache, miss → load from source → populate cache → return.
✓ Best Practice Standard pattern
Bad — mixing cache + business logic
public async Task<decimal>
  GetPriceAsync(int productId)
{
  // ⚠ cache logic tangled with business logic
  string key = "price_" + productId;
  if (_cache.TryGetValue(key,
    out decimal cached)) return cached;

  // business logic buried after cache logic
  var product = await _repo.GetAsync(productId);
  var price = product.BasePrice
    * (1 - product.Discount);

  _cache.Set(key, price,
    TimeSpan.FromSeconds(300));
  return price;
}
Fixed — clean cache-aside
// cache-aside using GetOrCreateAsync
public async Task<decimal>
  GetPriceAsync(int productId)
{
  return await _cache.
    GetOrCreateAsync(
      CacheKeys.Product(productId),
      async entry =>
      {
        // cache config — expiry + priority
        entry.AbsoluteExpirationRelativeToNow
          = CacheExpiry.Product;
        entry.Priority
          = CacheItemPriority.High;

        // business logic — clear and separated
        var product =
          await _repo.GetAsync(productId);
        return product.BasePrice
          * (1 - product.Discount);
      })!;
}
🔍 Cache-Aside flow
  • Check — look in cache. Hit → return immediately
  • Miss — call GetOrCreateAsync factory lambda
  • Load — fetch from DB/API inside the factory
  • Populate — result automatically stored with your expiry settings
  • Return — caller gets data regardless of cache hit or miss
Why GetOrCreateAsync beats manual TryGetValue
It handles the check-load-set pattern in one call. The factory is only invoked on a cache miss. Thread-safety is handled internally for simple cases. Use SemaphoreSlim only when you need per-key control.

5. Missing Cache Invalidation
Data updated in DB but cache serves stale version. Users see old prices, old stock levels, old profiles.
🔴 Critical Stale data served Write-Through fix
Bad — update without cache bust
public async Task
  UpdateProductAsync(Product product)
{
  // saves to DB
  await _repo.UpdateAsync(product);

  // ⚠ cache never invalidated
  // reads still return old data
  // _cache.Remove(...) MISSING!
}

public async Task
  DeleteProductAsync(int id)
{
  await _repo.DeleteAsync(id);
  // ⚠ deleted item still in cache!
}
Fixed — invalidate on every write
public async Task
  UpdateProductAsync(Product product)
{
  await _repo.UpdateAsync(product);

  // ✓ bust single entry
  _cache.Remove(
    CacheKeys.Product(product.Id));

  // ✓ also bust the list (it changed)
  _cache.Remove(
    CacheKeys.ProductList());
}

public async Task
  DeleteProductAsync(int id)
{
  await _repo.DeleteAsync(id);
  _cache.Remove(CacheKeys.Product(id));
  _cache.Remove(CacheKeys.ProductList());
}

// ✓ Advanced: tag-based invalidation
// group related entries under a tag
// bust the whole group with one call
var options = new MemoryCacheEntryOptions();
var tag = new CancellationTokenSource();
options.AddExpirationToken(
  new CancellationChangeToken(tag.Token));
// later: tag.Cancel() busts ALL tagged entries
🔍 Invalidation strategy by data type
  • Single entity updated → Remove(CacheKeys.Product(id))
  • List/collection changed → also Remove(CacheKeys.ProductList())
  • Bulk update → CancellationTokenSource tag bust
  • Time-sensitive data → short TTL + no manual invalidation
Rule: every write method must have a matching cache bust
Create → bust list cache. Update → bust entity + list. Delete → bust entity + list. If you add a write method without a bust, you will serve stale data.

6. Direct IMemoryCache — No Abstraction
IMemoryCache scattered across 10 classes — impossible to swap to Redis, impossible to mock in tests.
🔴 Critical DIP violation ICacheService fix
Bad — IMemoryCache injected everywhere
// ⚠ IMemoryCache in every service
public class ProductService
{
  private readonly IMemoryCache _cache;
  public ProductService(IMemoryCache c)
    => _cache = c;
}

public class UserService
{
  private readonly IMemoryCache _cache;
  // switch to Redis = edit EVERY class
}

// test: cannot mock IMemoryCache cleanly
// swap Redis: change 10+ classes
Fixed — ICacheService abstraction
// single abstraction in domain
public interface ICacheService
{
  Task<T?> GetAsync<T>(string key);
  Task      SetAsync<T>(string key, T val,
               TimeSpan? expiry = null);
  Task      RemoveAsync(string key);
}

// memory implementation
public class MemoryCacheService
  : ICacheService
{
  private static readonly TimeSpan Default
    = TimeSpan.FromMinutes(5);
  private readonly IMemoryCache _cache;

  public Task<T?> GetAsync<T>(string key)
  {
    _cache.TryGetValue(key, out T? v);
    return Task.FromResult(v);
  }
  public Task SetAsync<T>(string k, T v,
    TimeSpan? expiry)
  { _cache.Set(k, v, expiry ?? Default);
    return Task.CompletedTask; }
  public Task RemoveAsync(string k)
  { _cache.Remove(k);
    return Task.CompletedTask; }
}
// swap to Redis: class RedisCacheService : ICacheService
// test: new Mock<ICacheService>() — trivial
DI registration
builder.Services.AddMemoryCache();
builder.Services.AddScoped<ICacheService, MemoryCacheService>();
Swap to Redis: replace second line with RedisCacheService — zero other changes.

7. Blocking Async in Cache Methods
.Result / .Wait() inside cache methods cause deadlocks in ASP.NET Core.
🔴 Critical Deadlock risk Thread starvation
Bad — sync blocking async cache
public Product? GetProduct(int id)
{
  var key = CacheKeys.Product(id);

  // ⚠ .Result blocks the thread — deadlock!
  return _cache.GetOrCreateAsync(key,
    async e => await _repo.GetAsync(id))
    .Result;   // ← deadlock in ASP.NET Core
}

// also bad — sync method calling async repo
public List<Product> GetAll()
{
  return _repo.GetAllAsync()
    .GetAwaiter().GetResult();  // ← deadlock
}
Fixed — async all the way up
// ✓ async method — await all the way up
public async Task<Product?>
  GetProductAsync(int id)
{
  return await _cache.GetOrCreateAsync(
    CacheKeys.Product(id),
    async entry =>
    {
      entry.AbsoluteExpirationRelativeToNow
        = CacheExpiry.Product;
      // ✓ awaited — no thread blocking
      return await _repo.GetAsync(id);
    });
}

// ✓ controller uses async action
[HttpGet("{id}")]
public async Task<IActionResult>
  Get(int id)
{
  var product =
    await _service.GetProductAsync(id);
  return product is null
    ? NotFound() : Ok(product);
}
Never use .Result, .Wait(), .GetAwaiter().GetResult() on async cache methods in ASP.NET Core
The synchronization context causes deadlock. async/await must flow from the controller all the way down to the repository. No exceptions.

8. Generic GetOrSetAsync Helper
The same check-load-set pattern repeated in every method — extract to a reusable generic helper.
🟡 High DRY violation Extract Method fix
Bad — repeated in every method
// ⚠ same 5-line pattern in every service method
public async Task<Product?> GetProductAsync(int id)
{
  var key = CacheKeys.Product(id);
  if (_cache.TryGetValue(key, out Product? p))
    return p;
  p = await _repo.GetAsync(id);
  _cache.Set(key, p, CacheExpiry.Product);
  return p;
}

public async Task<User?> GetUserAsync(Guid id)
{
  // same 5 lines again...
}
Fixed — generic GetOrSetAsync
// reusable extension method
public static class CacheExtensions
{
  public static async Task<T?>
    GetOrSetAsync<T>(
      this IMemoryCache cache,
      string key,
      Func<Task<T?>> factory,
      TimeSpan? expiry = null)
    where T : class
  {
    if (cache.TryGetValue(key, out T? cached))
      return cached;

    var value = await factory();
    if (value is not null)
      cache.Set(key, value,
        expiry ?? TimeSpan.FromMinutes(5));
    return value;
  }
}

// usage — one clean line per method
public Task<Product?> GetProductAsync(int id)
  => _cache.GetOrSetAsync(
       CacheKeys.Product(id),
       () => _repo.GetAsync(id),
       CacheExpiry.Product);

public Task<User?> GetUserAsync(Guid id)
  => _cache.GetOrSetAsync(
       CacheKeys.UserProfile(id),
       () => _repo.GetUserAsync(id),
       CacheExpiry.UserSession);
DRY principle applied to caching
The check-load-set pattern is knowledge — it belongs in one place. The generic helper enforces consistent null-guarding (never cache null), consistent expiry defaults, and keeps every service method to a single expression.

Golden rules — memorise these
Rule 1
Never cache null
A null result means the item doesn't exist — don't fill the cache with nulls. Guard with if (value is not null).
Rule 2
Keys are code
Cache keys are a public contract. Put them in a static class. Include a version number so you can bust them globally.
Rule 3
Every write busts cache
Create → bust list. Update → bust entity + list. Delete → bust entity + list. No exceptions.
Rule 4
Abstract IMemoryCache
ICacheService in domain. MemoryCacheService in infra. Swap to Redis = one file change.
Rule 5
async all the way
Never .Result or .Wait() on a cache call inside ASP.NET Core. Deadlock guaranteed.
Rule 6
Lock on miss
High-traffic cache miss = stampede. Use SemaphoreSlim per key or GetOrCreateAsync built-in locking.
