1
SQL Injection → Parameterized Queries
String interpolation in SQL = critical security vulnerability. Any user input passed directly into a query string can delete your database.
🔴 Critical — Security DIP adjacent
Bad — SQL injection risk
public void SaveInvoice(Invoice inv)
{
  var conn = new SqlConnection(
    "Server=localhost;Password=1234");
  conn.Open();

  // ⚠ string interpolation = injection
  var sql = $"INSERT INTO Invoices "
          + $"VALUES ('{inv.Customer}',"
          + $"'{inv.Amount}', '{inv.Date}')";
  conn.Execute(sql);
}
Fixed — parameterized query
public async Task SaveInvoiceAsync(Invoice inv)
{
  // connection string from IConfiguration
  await using var conn = new SqlConnection(
    _config.GetConnectionString("Default"));

  // ✓ named parameters — injection impossible
  const string sql =
    "INSERT INTO Invoices "
    "(Customer, Amount, Date) "
    "VALUES (@Customer, @Amount, @Date)";

  await conn.ExecuteAsync(sql, new
  {
    inv.Customer,
    inv.Amount,
    inv.Date
  });
}
🔍 How to find it
  • Search: $"... {variable} ..." inside any SQL string
  • Search: "SELECT" + variable or "INSERT" + variable
  • Any Execute() / Query() call with a built string, not a const
  • No @ParameterName in the SQL and no anonymous object passed as second arg
Rule
SQL strings must be const string — never built with interpolation or concatenation. User input goes in the parameter object only. Dapper's @Name syntax handles escaping automatically.
2
Tight Coupling → IInvoiceRepository + IEmailSender Injected
Business logic directly instantiates SqlConnection, SmtpClient — impossible to test, impossible to swap. DIP broken.
🔴 Critical — DIP SRP
Bad — hardcoded dependencies
public class InvoiceService
{
  // ⚠ new = tightly coupled
  private readonly SqlConnection _db
    = new SqlConnection("...");
  private readonly SmtpClient _smtp
    = new SmtpClient("smtp.host");

  // cannot mock, cannot test in isolation
}
Fixed — injected interfaces
// define ports in domain layer
public interface IInvoiceRepository
{
  Task SaveAsync(Invoice invoice);
  Task<Invoice?> GetByIdAsync(int id);
}
public interface IEmailSender
{
  Task SendAsync(string to,
    string subject, string body);
}

// ✓ depends on abstractions only
public class InvoiceService
{
  private readonly IInvoiceRepository _repo;
  private readonly IEmailSender _email;

  public InvoiceService(
    IInvoiceRepository repo,
    IEmailSender email)
  { _repo = repo; _email = email; }
}
// Program.cs:
// .AddScoped<IInvoiceRepository,
//            SqlInvoiceRepository>()
// .AddScoped<IEmailSender, SmtpEmailSender>()
🔍 How to find it
  • Search: = new ConcreteClass() as a field initializer
  • Search: new SqlConnection / new SmtpClient / new HttpClient inside class bodies
  • Constructor has zero parameters but class does I/O
  • Cannot write a unit test without real infrastructure running
Principle: DIP — Dependency Inversion
High-level modules (InvoiceService) must not depend on low-level modules (SqlConnection). Both depend on abstractions (IInvoiceRepository). Interfaces live in the domain layer. Concrete classes live in the infrastructure layer.
3
SRP Violation → Validation / Pricing / Persistence / Notification Split
One class doing four jobs. Every change to pricing, DB schema, email template, or validation rules touches the same file.
🔴 Critical — SRP Extract Class
Bad — one method, four concerns
public class InvoiceManager  // ⚠
{
  public void Process(Invoice inv)
  {
    // 1. validate
    if (inv.Amount <= 0)
      throw new Exception("bad");

    // 2. calculate price
    inv.Tax = inv.Amount * 0.07m;
    inv.Total = inv.Amount + inv.Tax;

    // 3. save to DB
    var conn = new SqlConnection(...);
    conn.Execute($"INSERT...");

    // 4. send email
    new SmtpClient().Send(...);
  }
}
Fixed — each class has one job
// 1. validation only
public class InvoiceValidator
{
  public void Validate(Invoice inv)
  {
    if (inv.Amount <= 0)
      throw new ArgumentException(
        "Amount must be positive");
  }
}
// 2. pricing only
public class InvoicePricer
{
  public void ApplyTax(Invoice inv)
  {
    inv.Tax   = inv.Amount * TaxRate;
    inv.Total = inv.Amount + inv.Tax;
  }
}
// 3. persistence → IInvoiceRepository
// 4. notification → IEmailSender

// orchestrator — no logic, just coordination
public class InvoiceService
{
  public async Task ProcessAsync(Invoice inv)
  {
    _validator.Validate(inv);
    _pricer.ApplyTax(inv);
    await _repo.SaveAsync(inv);
    await _email.SendAsync(...);
  }
}
🔍 How to find it
  • Ask: "who would ask me to change this class?" — PM (logic), DBA (SQL), designer (email) = 3 owners = SRP broken
  • Method contains comment blocks labelling different concerns (// validate, // save, // email)
  • Class name ends in Manager, Processor, Handler
  • File is over 150 lines or method over 30 lines
Principle: SRP — Single Responsibility
A class has one reason to change. Split by concern: Validator changes when rules change. Pricer changes when tax rates change. Repository changes when DB schema changes. Mailer changes when email template changes. Each class has exactly one owner.
4
Magic Literals → Constants + Configuration
0.07m, 30, "no-reply@app.com" scattered inline — no one knows what they mean. Change requires grep across the whole codebase.
🟡 High — Readability DRY
Bad — magic everywhere
public void Process(Invoice inv)
{
  inv.Tax = inv.Amount * 0.07m;   // what rate?
  inv.DueDate = inv.Date.AddDays(30);  // why 30?
  if (inv.Amount > 10000)            // what is 10000?
    _email.Send("boss@co.com", ...);
  _email.From = "no-reply@app.com";
}
Fixed — named constants + config
// constants — single source of truth
private const decimal TaxRate          = 0.07m;
private const int     PaymentTermsDays = 30;
private const decimal LargeInvoiceLimit= 10_000m;

// config-driven — from appsettings.json
public class InvoiceSettings
{
  public string FromEmail    { get; set; } = "";
  public string AlertEmail   { get; set; } = "";
}

public void Process(Invoice inv)
{
  inv.Tax     = inv.Amount * TaxRate;
  inv.DueDate = inv.Date.AddDays(PaymentTermsDays);
  if (inv.Amount > LargeInvoiceLimit)
    _email.Send(_settings.AlertEmail, ...);
}
// appsettings.json:
// "Invoice": { "FromEmail":"no-reply@co.com",
//              "AlertEmail":"boss@co.com" }
🔍 How to find it
  • Any decimal/int literal that is not 0 or 1 inside business logic
  • Literal strings for emails, roles, status values, queue names
  • Same number appears in 2+ places
  • You need a comment to explain what a number means
Rule: Two categories
Business constants (TaxRate, PaymentTermsDays) → private const in the class. Environment config (email addresses, URLs, keys) → appsettings.json bound via IOptions<InvoiceSettings>. Never hardcode environment-specific values.
5
Duplicated DB Block → Single SaveInvoice() Call
The same connection + query block copy-pasted in multiple methods. DRY broken — one bug means multiple fixes.
🟡 High — DRY Extract Method
Bad — duplicated connection block
public void CreateInvoice(Invoice inv)
{
  var conn = new SqlConnection("...");
  conn.Open();
  conn.Execute("INSERT...", inv);
  conn.Close();  // duplicated block #1
}

public void UpdateInvoice(Invoice inv)
{
  var conn = new SqlConnection("...");
  conn.Open();
  conn.Execute("UPDATE...", inv);
  conn.Close();  // duplicated block #2
}
// add method = copy-paste block again
Fixed — single repository method
// all DB logic in one place
public class SqlInvoiceRepository
  : IInvoiceRepository
{
  private readonly string _conn;
  public SqlInvoiceRepository(IConfiguration cfg)
    => _conn = cfg.GetConnectionString("Default")!;

  public async Task SaveAsync(Invoice inv)
  {
    await using var db = new SqlConnection(_conn);
    const string sql =
      "INSERT INTO Invoices (Customer,Amount,Date)"
      "VALUES (@Customer,@Amount,@Date)";
    await db.ExecuteAsync(sql, new
      { inv.Customer, inv.Amount, inv.Date });
  }
}

// caller — one line, no duplication
await _repo.SaveAsync(invoice);
🔍 How to find it
  • Search: new SqlConnection( — if it appears more than once, extract it
  • Copy-pasted blocks with only the SQL string changed
  • Same conn.Open() / conn.Execute() / conn.Close() pattern repeated
  • Bug fix in one block that also exists in another block
Principle: DRY — Don't Repeat Yourself
Every piece of knowledge has a single authoritative representation. The DB connection setup is knowledge — it belongs in one place (SqlInvoiceRepository). Callers only call SaveAsync(). If the schema changes, one method changes.
6
Untestable Design → All I/O Behind Interfaces
Code that requires a real database, SMTP server, or Stripe account to run any test is untestable in isolation. Every dependency must be mockable.
🔴 Critical — Testability DIP
Bad — impossible to unit test
public class InvoiceService
{
  public void Process(Invoice inv)
  {
    // needs real DB
    var conn = new SqlConnection(...);
    conn.Execute(...);
    // needs real SMTP
    new SmtpClient().Send(...);
    // needs real Stripe
    new StripeClient("sk_live").Charge();
  }
}

// test is impossible without infra:
// [Test] void Process_ValidInvoice_SavesAndSends()
// { ??? } // need real DB running
Fixed — fully testable with Moq
// all I/O behind interfaces
public class InvoiceService
{
  private readonly IInvoiceRepository _repo;
  private readonly IEmailSender       _email;
  public InvoiceService(IInvoiceRepository r,
    IEmailSender e) { _repo=r; _email=e; }

  public async Task ProcessAsync(Invoice inv)
  {
    await _repo.SaveAsync(inv);
    await _email.SendAsync(inv.Email,
      "Invoice", $"Amount: {inv.Total:C}");
  }
}

// ✓ full unit test — no infra needed
[Test]
public async Task
  Process_ValidInvoice_SavesAndSendsEmail()
{
  // Arrange
  var repo  = new Mock<IInvoiceRepository>();
  var email = new Mock<IEmailSender>();
  var svc   = new InvoiceService(
    repo.Object, email.Object);
  var inv   = new Invoice { Amount=100, ... };

  // Act
  await svc.ProcessAsync(inv);

  // Assert
  repo.Verify(r => r.SaveAsync(inv),
    Times.Once());
  email.Verify(e => e.SendAsync(
    inv.Email, It.IsAny<string>(),
    It.IsAny<string>()),
    Times.Once());
}
🔍 How to find it (the testability test)
  • Ask: "Can I run this test without a DB / SMTP / Stripe?" — if no, design is untestable
  • Any new ConcreteInfra() inside the class = cannot be mocked
  • Static method calls to infrastructure (File.WriteAllText, Console.Write) = untestable
  • No interfaces = Moq has nothing to mock
Testability Rule
If you can write new Mock<IDependency>() for every external concern and run the test in memory with no network — the design is correct. Testability is a consequence of DIP: depend on interfaces, inject them, mock them in tests.
