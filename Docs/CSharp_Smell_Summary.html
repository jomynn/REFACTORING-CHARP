Code Smell Summary
Before Refactoring

Every common smell you will find in a C# refactoring test — categorised by SOLID principle, with the bad code pattern, what to grep for, and the fix pattern name.

Critical — fix first
High — fix in same session
Medium — fix if time allows
S — Single Responsibility Principle
🏛
God Class
SRP · Single Responsibility
🔴 Critical S
Bad code
public class UserManager { // validate + save DB + send email + generate PDF public void Process(User user) { // 150+ lines of mixed concerns } }
  • File over 200 lines — near-certain SRP violation
  • Class name ends in Manager, Handler, Helper, Utils
  • Constructor takes 5+ dependencies
  • Multiple private method groups with unrelated concerns
✓ Fix: Extract Class — Service / Repository / Mailer
🔀
Mixed Abstraction Levels
SRP · Single Responsibility
🔴 Critical S
Bad code
public void PlaceOrder(Order order) { // business rule if (order.Total <= 0) throw...; // raw SQL inside business logic var conn = new SqlConnection("..."); conn.Execute("INSERT..."); // SMTP inside business logic new SmtpClient().Send(...); }
  • Business logic and SQL/HTTP/SMTP in same method
  • Method does "what to do" and "how to do it technically"
  • Function longer than 20–30 lines
✓ Fix: Extract Method + separate infra classes
O — Open / Closed Principle
🔀
if / else Chain on Type
OCP · Open/Closed
🔴 Critical O
Bad code
public decimal GetDiscount(string type) { if (type == "VIP") return 0.30m; else if (type == "Member") return 0.15m; else if (type == "New") return 0.05m; // add type = EDIT THIS FILE return 0m; }
  • Search: if (x == "...") or switch(type) in service classes
  • Every new business type forces editing this method
  • Same switch duplicated in multiple methods
✓ Fix: Strategy pattern — one class per type
🎛
switch on Action / Status
OCP · Open/Closed
🔴 Critical O
Bad code
switch (order.Status) { case "Pending": ValidateOrder(); break; case "Confirmed": ChargePayment(); break; case "Paid": ShipOrder(); break; // add status = edit THIS + other methods }
  • Same switch in multiple methods (Process, GetLabel, GetIcon)
  • Adding new status → shotgun surgery across files
  • Status logic scattered, not encapsulated
✓ Fix: State pattern — each status owns its transitions
L — Liskov Substitution Principle
🔍
instanceof / is T Check
LSP · Liskov Substitution
🔴 Critical L
Bad code
public void Draw(Shape shape) { if (shape is Circle c) DrawCircle(c.Radius); else if (shape is Rectangle r) DrawRect(r.Width, r.Height); // caller must know concrete type — LSP broken }
  • Search: is ConcreteType or as ConcreteType in non-factory code
  • Caller checks type before using object
  • Base type doesn't expose what callers need
✓ Fix: Add abstract method to base — shape.Draw()
💥
NotImplementedException in Subclass
LSP · Liskov Substitution
🔴 Critical L
Bad code
public class ReadOnlyList : List { public override void Add(T item) => throw new NotImplementedException(); public override void Remove(T item) => throw new NotSupportedException(); }
  • Search: throw new NotImplementedException()
  • Search: throw new NotSupportedException()
  • Subclass breaks contract of the base class
  • Square / Rectangle anti-pattern
✓ Fix: Split into IReadable / IMutable interfaces
I — Interface Segregation Principle
🍔
Fat Interface
ISP · Interface Segregation
🟡 High I
Bad code
public interface IAnimalActions { void Eat(); void Sleep(); void Fly(); // fish can't fly void Bark(); // cats don't bark void Swim(); // dogs can't swim well }
  • Interface has more than 5–7 methods
  • Classes implement the interface but leave empty or throw stubs
  • Interface name is too generic: IService, IManager, IHelper
  • You mock half the interface just to test one feature
✓ Fix: Role interfaces — IFlyable, ISwimmable, IBarkable
👻
Empty Stub Methods
ISP · Interface Segregation
🟡 High I
Bad code
public class RobotWorker : IWorker { public void Work() { /* ok */ } public void Eat() { } // forced empty stub public void Sleep() { } // forced empty stub public void Bark() => throw new NotImplementedException(); }
  • Search: { } (empty body) on interface method implementations
  • Search: throw new NotImplementedException
  • Class forced to implement methods it logically cannot do
✓ Fix: Split IWorkable / IFeedable / ISleepable
D — Dependency Inversion Principle
🔗
new ConcreteClass() in Constructor
DIP · Dependency Inversion
🔴 Critical D
Bad code
public class OrderService { private readonly SqlOrderRepository _repo = new SqlOrderRepository(); private readonly SmtpMailer _mailer = new SmtpMailer(); // tightly coupled — cannot swap, test, or mock }
  • Search: new ConcreteService() inside any class field or constructor
  • Cannot unit test without spinning up real DB / SMTP
  • Swapping SQL → EF Core requires editing business logic
✓ Fix: Constructor injection — IOrderRepository, IMailer
📦
Business Layer Imports Infrastructure
DIP · Dependency Inversion
🔴 Critical D
Bad code
// in Domain/OrderService.cs using Dapper; // DB driver using System.Net.Mail; // SMTP using Twilio.Rest.Api; // SMS SDK using Stripe; // Payment SDK // domain layer knows too much about infra
  • Domain/service file has using statements for DB, SMTP, HTTP SDK
  • Dependency arrow points outward (domain → infra) — wrong direction
  • Changing payment provider → editing business logic
✓ Fix: Ports & Adapters — interfaces in domain, SDKs in infra
🔑
Hardcoded Credentials / Connection Strings
DIP + Security
🔴 Critical D
Bad code
var conn = new SqlConnection( "Server=prod;Password=P@ss1234" ); var stripe = new StripeClient( "sk_live_REAL_KEY_HERE" );
  • Search: "Server=, Password=, sk_live_, ACxxxx (Twilio)
  • Credentials committed to source control — security risk
  • Cannot change env without recompiling
✓ Fix: IConfiguration + appsettings.json + IOptions<T>
General Code Smells — apply to any principle
🔢
Magic Numbers & Strings
Readability · DRY
🟡 High DRY
Bad code
if (score > 85) grade = "A"; if (attempts > 5) LockAccount(); expiry = DateTime.Now.AddSeconds(86400); if (role == "admin") Grant(); // what does 85, 5, 86400 mean?
  • Any literal number that isn't 0 or 1 in logic
  • Literal strings used for roles, status, types
  • Same number repeated in 3+ places
✓ Fix: private const int GradeAThreshold = 85;
⬇️
Long Method / Arrow Code
Readability · SRP
🟡 High S
Bad code — 5 levels deep
if (user != null) { if (user.IsActive) { if (user.Age >= 18) { if (request != null) { // actual logic buried here }} }} }
  • Method over 20–30 lines
  • Nesting deeper than 3 levels (arrow / pyramid of doom)
  • Multiple return paths buried deep in nesting
  • Hard to write a single unit test that covers the method
✓ Fix: Guard clauses + Extract Method
🏷
Poor Naming
Readability
🟡 High General
Bad code
public decimal Do(decimal x, int n) { var r = 0m; // what is r? var tmp = x * 0.07m; // what is tmp? return r + tmp; }
  • Single-letter params: x, n, d, r, tmp
  • Method name describes HOW not WHAT: Do(), Process(), Handle()
  • Boolean params: Process(user, true, false, null)
  • You need a comment to explain what a variable does
✓ Fix: Rename — CalculateFinalPrice(orderTotal, years)
💉
SQL Injection Risk
Security + DIP
🔴 Critical Security
Bad code
conn.Execute( $"INSERT INTO Users VALUES ('{name}', '{email}')" ); // string interpolation in SQL = injection risk
  • Search: $"SELECT ... {variable}" — interpolation in SQL
  • Search: "... + variable + ..." — concatenation in SQL
  • No @ParameterName or SqlParameter usage
✓ Fix: Parameterized query — new { Name = name }
💀
Dead / Commented-out Code
YAGNI · Readability
🔵 Medium YAGNI
Bad code
// var old = GetOldUser(id); // if (old != null) return old.Map(); public void OldMethod() { /* never called */ } // TODO: fix this later (from 2021)
  • Blocks of commented-out code — use git history instead
  • Unused methods, fields, or classes — IDE warnings
  • TODO comments older than current sprint
✓ Fix: Delete it — git is your backup
📢
Console.WriteLine as Logging
DIP · Observability
🟡 High D
Bad code
Console.WriteLine("Processing: " + id); Console.WriteLine("ERROR: " + ex.Message); // not routable, no severity, no sink, no structure
  • Search: Console.WriteLine anywhere in service classes
  • String concatenation in log messages — no named properties
  • No log levels — everything treated as equal severity
✓ Fix: ILogger<T> injection + _logger.LogInformation("{Id}", id)
