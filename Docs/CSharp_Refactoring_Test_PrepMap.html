SOLID Principles
Foundation — every senior test includes this
Must Know Critical
PRIORITY: 10 / 10
  • S — Extract class, one reason to change
  • O — Strategy / Policy, open for extension
  • L — Interface over deep inheritance
  • I — Role interfaces, small and focused
  • D — Constructor injection, depend on abstraction
  • Name the pattern when you apply it — e.g. "This is Strategy"

Clean Architecture Layers
Required for Senior → Lead level
Must Know Critical
PRIORITY: 9.5 / 10
  • Domain — Entities, Value Objects, Domain Events
  • Application — Use Cases, CQRS, MediatR handlers
  • Infrastructure — DB, SMTP, HTTP, Queue adapters
  • Presentation — Controllers, minimal API, DTOs
  • Dependency rule: arrows always point inward
  • Ports & Adapters (Hexagonal) — know the pattern name

Design Patterns (GoF)

  • Factory / Factory Method — object creation

Unit Testing with Moq + NUnit/xUnit
Senior must write tests, not just code
Must Know Critical
PRIORITY: 9 / 10
  • Arrange / Act / Assert structure every test
  • Moq — Mock<T>, Setup(), Verify(), Returns()
  • xUnit — [Fact], [Theory], [InlineData]
  • NUnit — [Test], [TestCase], Assert.That()
  • Test one thing per test — single reason to fail
  • Mock only direct dependencies, not transitive ones
  • Name: MethodName_Scenario_ExpectedResult

Refactoring Patterns (Fowler)
Named moves — call them by name in the test
Must Know High
PRIORITY: 8.5 / 10
  • Extract Method — long method → named methods
  • Extract Class — SRP violation → split class
  • Replace Magic Number with Constant
  • Replace Conditional with Polymorphism
  • Introduce Parameter Object — 5+ params
  • Guard Clauses — flatten nested if pyramid
  • Move Method — method to the class it belongs
  • Rename — x, tmp → descriptive names

Code Smell Identification
Scan fast — first 5–10 min of the test
Must Know High
PRIORITY: 8.5 / 10
  • God class — file over 200 lines, *Manager name
  • Long method — over 20–30 lines, multiple concerns
  • Magic numbers/strings — 0.15, 86400, "admin"
  • Data clumps — same 4 params passed everywhere
  • Feature Envy — method uses another class's data more
  • Shotgun surgery — 1 change hits 10 files
  • Primitive obsession — string for email, int for money
  • Dead code — commented-out blocks, unused methods
📨
CQRS + MediatR
Expected at Lead level in .NET
Lead Level High
PRIORITY: 7.5 / 10
  • IRequest<T> — command or query object
  • IRequestHandler<,> — one handler per request
  • INotification — domain events / fan-out
  • IPipelineBehavior — cross-cutting: logging, validation
  • Command = write, no return; Query = read, return data
  • How it relates to SRP and OCP

Domain Model Concepts
Shows architectural maturity at Lead level
Lead Level Medium
PRIORITY: 7 / 10
  • Entity — identity-based, mutable
  • Value Object — equality by value, immutable
  • Aggregate Root — consistency boundary
  • Domain Event — something significant happened
  • Repository interface — defined in domain, not infra

Async / Await Patterns
Common in refactoring — sync → async
Must Know Medium
PRIORITY: 7 / 10
  • Never .Result or .Wait() — causes deadlock
  • Always await all the way up — no sync-over-async
  • ConfigureAwait(false) in library code
  • Task.WhenAll() — parallel async calls
  • CancellationToken — pass through every async method
  • IAsyncDisposable — async using pattern
🚨
Exception Handling Patterns
Often broken in code under test
Common Smell Medium
PRIORITY: 6.5 / 10
  • Never catch Exception and swallow silently
  • Custom domain exceptions — OrderNotFoundException
  • Global exception middleware in ASP.NET Core
  • ArgumentException vs InvalidOperationException
  • Result pattern — return errors as values, not throws
  • Never use exceptions for flow control
📋
Structured Logging
Console.WriteLine is always a smell
Common Fix Medium
PRIORITY: 6 / 10
  • ILogger<T> injected — never static or Console
  • Named placeholders: {OrderId} not string concat
  • Correct log levels: Info / Warning / Error / Critical
  • Serilog — enrichers, sinks, structured output
  • Never log sensitive data — PII, passwords, tokens
60-minute time-box strategy
Minute-by-minute plan
Senior/Lead — they judge thinking speed AND architectural decisions
TimePhaseWhat to doSenior tip
0–5 min Read Read all instructions fully. Note language, framework, any explicit constraints. Do NOT touch code. Check if tests exist — if yes, run them first so you know baseline.
5–15 min Explore Apply SOLID checklist. List the top 3–5 smells with file + line reference. Write them as comments. Identify the architecture violation first (DIP/SRP) — structural fixes unlock everything else.
15–20 min Plan Write a 3-line refactoring plan as a comment block at top of file. Prioritize: Structural → Readability → Polish. This comment block shows the reviewer your Lead-level thinking even if you don't finish.
20–45 min Refactor Start with the biggest structural issue. Run tests after each completed pattern. Keep each change small and verifiable. Name the pattern out loud (in comments): "// Strategy pattern — each discount type is a class"
45–55 min Test Write or fix unit tests. At least 1 test per extracted class. AAA structure. Mock all external dependencies. A refactor without tests is incomplete at Lead level — reviewers expect Moq usage.
55–60 min Review Re-read your changes. Check naming, no magic numbers remain, no Console.WriteLine, tests green. Add a 3-line summary comment: what you changed, why, and what you would do next with more time.
⏱ If time runs short — triage rule
Priority order when you can't finish everything
1st — Fix DIP violations
Extract interfaces, inject via constructor. This unlocks testability.
2nd — Fix SRP violations
Split god classes. One class, one job. Most visible to reviewer.
3rd — Replace switch/if chains
Strategy pattern. Shows OCP knowledge clearly.
4th — Extract methods + naming
Guard clauses, named constants, descriptive names.
5th — Write tests
Even 2–3 Moq tests signal Lead-level discipline.

Likely interview questions for Senior → Lead
Architecture
"What's the difference between Clean Architecture and MVC?"
→ Dependency direction, layer isolation, testability without framework
SOLID
"Can you violate DRY while following SOLID?"
→ Yes — a bad abstraction is worse than duplication (AHA principle)
Patterns
"When would you NOT use the Repository pattern?"
→ Simple CRUD apps, Event Sourcing stores, direct Dapper usage
Testing
"What's the difference between a mock and a stub?"
→ Stub returns data; mock verifies behaviour was called
Design
"How do you handle cross-cutting concerns without violating SRP?"
→ Decorator, Middleware, IPipelineBehavior in MediatR
Lead thinking
"How would you introduce clean architecture to a legacy codebase?"
→ Strangler Fig pattern, seam injection, migrate module by module
Trade-offs
"When is a God class acceptable?"
→ Scripts, prototypes, or when the domain truly is one concern
Process
"How do you refactor safely without breaking production?"
→ Tests first, small commits, feature flags, refactor ≠ rewrite
